# Architecture Overview

## System Overview

The AI-Assisted Journal System is a full-stack application that allows users to write journal entries after immersive nature sessions and receive AI-generated emotional insights.

The system is designed with a **modular architecture** separating frontend, backend, database, and AI processing layers. This separation improves maintainability, scalability, and performance.

### High Level Architecture

```
Frontend (React + Vite)
        │
        │ HTTP API (Axios)
        ▼
Backend API (Node.js + Express)
        │
        ├── MongoDB Database
        │
        └── LLM Service (Gemini API)
```

# Backend Architecture

The backend follows a **layered architecture**:

```
Routes → Controllers → Services → Database
```

### Routes

Define API endpoints and delegate requests to controllers.

```
POST /api/journal
GET /api/journal/:userId
POST /api/journal/analyze
GET /api/journal/insights/:userId
```

### Controllers

Handle HTTP logic such as request validation and response formatting.

### Services

Contain business logic such as:

* LLM emotion analysis
* insights generation

### Database Layer

MongoDB stores journal entries using a Mongoose schema.

Example Journal Schema:

```
{
 userId: String,
 ambience: String,
 text: String,
 emotion: String,
 keywords: [String],
 summary: String,
 createdAt: Date
}
```

Indexes are used on `userId` to improve query performance.

# Data Flow

### Creating a Journal Entry

1. User writes a journal entry in the React UI.
2. Axios sends a POST request to the backend.
3. Express controller validates the request.
4. Journal entry is stored in MongoDB.

```
User → React → Express API → MongoDB
```

---

### Emotion Analysis

1. User clicks "Analyze".
2. Backend sends journal text to the LLM.
3. Gemini API returns emotion insights.
4. Backend parses and returns structured JSON.

```
User → React → Express → Gemini API → Express → React
```

---

### Insights Generation

Insights are generated by aggregating stored journal entries.

Metrics include:

* total entries
* most common emotion
* most used ambience
* frequently used keywords

# Scaling to 100k Users

To support large scale usage, the system can be scaled using several strategies.

### Horizontal Backend Scaling

Deploy multiple backend instances behind a load balancer.

Example architecture:

```
Load Balancer
   │
   ├── API Server 1
   ├── API Server 2
   └── API Server 3
```

This distributes incoming requests and improves reliability.

### Database Scaling

Possible approaches:

* MongoDB Atlas cluster
* read replicas
* sharding for large datasets

Indexes on frequently queried fields such as `userId` ensure fast query performance.

### CDN for Frontend

The frontend can be deployed to a CDN (e.g., Vercel, Netlify) to reduce latency for global users.

# Reducing LLM Cost

LLM requests can become expensive at scale. Several strategies can reduce cost.

### Result Caching

If a journal entry has already been analyzed, the result can be reused.

Example strategy:

```
hash(journal_text) → cached_analysis
```

If the same text appears again, the cached result is returned instead of calling the LLM.

### Rate Limiting

Limit analysis requests per user to prevent abuse.

Example:

```
5 analysis requests per minute per user
```

### Batch Processing

For high traffic systems, journal analysis could be processed asynchronously using a job queue.

Example tools:

* BullMQ
* RabbitMQ
* Kafka

---

# Caching Repeated Analysis

To avoid repeated LLM calls for identical inputs, a caching mechanism can be implemented.

Example flow:

```
User submits journal
        │
Check cache (Redis / DB)
        │
If cached → return result
If not → call LLM
        │
Store result in cache
```

Redis is ideal for this purpose due to its fast read performance.

Benefits:

* reduced latency
* lower API costs
* better scalability

# Protecting Sensitive Journal Data

Journal entries may contain personal or emotional information, so security is important.

### Data Encryption

Sensitive data should be encrypted in transit using HTTPS.

### Authentication

User authentication (JWT or OAuth) ensures that only authorized users can access their journals.

### Access Control

Users should only be able to access their own journal entries.

Example check:

```
journal.userId === authenticatedUser.id
```

### Data Privacy

Logs should avoid storing sensitive journal text.
